# Phase 02.2: Crime Cluster UX - Research

**Researched:** 2026-03-23
**Domain:** Mapbox clustering, heatmap layers, severity weighting, tap interactions (iOS/Swift)
**Confidence:** HIGH

## Summary

This phase replaces grid-based crime markers with incident-based Mapbox native clustering, adds severity weighting, category icons, tap-to-detail interaction, and reworks the heatmap gradient. The existing codebase already uses `@_spi(Experimental) import MapboxMaps` with declarative `MapContentBuilder` and Mapbox v11.20.1, so all required APIs are available.

The main work divides into four tracks: (1) refactor the GeoJSON source from `DensityGrid` cells to raw `CrimeIncident` coordinates with severity properties, (2) rework heatmap gradient to severity-weighted green-to-red, (3) replace cluster layer styling with severity-aware coloring and category icons at high zoom, and (4) add tap interaction with bottom sheet crime list.

**Primary recommendation:** Build a single `crimeIncidentFC` feature collection from raw `CrimeIncident` data with severity weight and category properties, feed it to both the heatmap source and the cluster source, and use Mapbox `TapInteraction(.layer(...))` for cluster/individual tap handling.

## Architecture Patterns

### Data Flow for Crime Layer

```
CrimeService.incidents: [CrimeIncident]
    |
    v
HFMapView.crimeIncidentFC (computed property)
    |-- severity weight property (3x/2x/1.5x/1x based on category)
    |-- category property ("violent"/"property"/"vehicle"/"other")
    |-- description, date properties (for detail sheet)
    |
    +---> GeoJSONSource "crime-incidents" (cluster: false)
    |         |
    |         +---> HeatmapLayer "crime-heat" (heatmapWeight from severity)
    |
    +---> GeoJSONSource "crime-clusters" (cluster: true, clusterRadius: 50, clusterMaxZoom: 14)
              |
              +---> clusterProperties: { "severity_sum": accumulated sum of "weight" }
              |
              +---> CircleLayer "crime-cluster-circles" (filter: has point_count)
              +---> SymbolLayer "crime-cluster-labels" (shows severity_sum)
              +---> SymbolLayer "crime-single-icons" (filter: NOT has point_count, shows category icon)
```

### Pattern 1: Severity Classification from Category String

**What:** Map crime category strings (from SODA API) to severity tiers using the existing `CrimeType` enum pattern.
**When to use:** When building the GeoJSON feature collection from incidents.

```swift
// Extend existing CrimeType or add severity weight computation
enum CrimeSeverity {
    case violent    // weight 3.0
    case property   // weight 2.0
    case vehicle    // weight 1.5
    case other      // weight 1.0

    static func from(category: String) -> CrimeSeverity {
        let lower = category.lowercased()
        // Violent
        if lower.contains("homicide") || lower.contains("assault") ||
           lower.contains("robbery") || lower.contains("rape") ||
           lower.contains("weapon") || lower.contains("kidnap") {
            return .violent
        }
        // Property
        if lower.contains("burglary") || lower.contains("theft") ||
           lower.contains("arson") || lower.contains("larceny") ||
           lower.contains("stolen") || lower.contains("shoplifting") {
            // Distinguish vehicle theft
            if lower.contains("vehicle") || lower.contains("auto") || lower.contains("car") {
                return .vehicle
            }
            return .property
        }
        // Vehicle
        if lower.contains("vehicle") || lower.contains("auto") || lower.contains("car") {
            return .vehicle
        }
        return .other
    }

    var weight: Double {
        switch self {
        case .violent:  return 3.0
        case .property: return 2.0
        case .vehicle:  return 1.5
        case .other:    return 1.0
        }
    }

    var iconName: String {
        switch self {
        case .violent:  return "bolt.shield.fill"      // weapon/danger
        case .property: return "house.fill"             // property
        case .vehicle:  return "car.fill"               // vehicle
        case .other:    return "circle.fill"             // generic
        }
    }
}
```

### Pattern 2: Cluster Properties for Severity Aggregation

**What:** Use `clusterProperties` to aggregate severity-weighted sum across clustered points.
**When to use:** When configuring the GeoJSONSource for clusters.

```swift
var src = GeoJSONSource(id: "crime-clusters")
src.cluster = true
src.clusterRadius = 50
src.clusterMaxZoom = 14
src.clusterProperties = [
    "severity_sum": Exp(.accumulated) {
        Exp(.sum) {}
        Exp(.get) { "weight" }
    }
]
```

Each GeoJSON feature must include a `"weight"` property with the severity weight value. Clustered features will then have a `"severity_sum"` property containing the accumulated weighted total.

### Pattern 3: TapInteraction for Cluster and Individual Taps

**What:** Use Mapbox `TapInteraction(.layer(...))` to handle taps on cluster circles and individual markers.
**When to use:** In the Map body alongside existing gesture handlers.

```swift
// Inside Map {} body:
TapInteraction(.layer("crime-cluster-circles")) { feature, context in
    // feature.properties contains "severity_sum", "point_count", "cluster_id"
    // Use cluster_id to query leaves for the detail sheet
    handleClusterTap(feature: feature)
    return true
}

TapInteraction(.layer("crime-single-icons")) { feature, context in
    // feature.properties contains individual crime data
    handleSingleCrimeTap(feature: feature)
    return true
}
```

**Important:** `TapInteraction` requires the layer ID (string), not a source ID. It fires for the topmost matching feature. Return `true` to stop propagation.

### Pattern 4: Querying Cluster Leaves for Detail Sheet

**What:** Use `MapboxMap.getGeoJsonClusterLeaves` to get all individual crimes within a tapped cluster.
**When to use:** When user taps a cluster circle.

```swift
// Access via MapReader proxy:
proxy.map?.getGeoJsonClusterLeaves(
    forSourceId: "crime-clusters",
    feature: clusterFeature,
    limit: 100,
    offset: 0
) { result in
    switch result {
    case .success(let leaves):
        // leaves is FeatureCollection — extract crime details from properties
        let crimes = leaves.features.compactMap { f -> CrimeDetail? in
            guard let cat = f.properties?["category"]?.string,
                  let desc = f.properties?["description"]?.string else { return nil }
            return CrimeDetail(category: cat, description: desc, ...)
        }
        // Show bottom sheet with crimes list
    case .failure(let error):
        AppLogger.map.error("Failed to get cluster leaves: \(error)")
    }
}
```

### Pattern 5: Category Icons via SymbolLayer with Image Expressions

**What:** Register SF Symbols as map images and use `iconImage` expression to select by category.
**When to use:** For individual (unclustered) crime markers at z15+.

Two approaches:
1. **SF Symbols as map images**: Register images in `makeUIView` or via `StyleImage`, then reference by name in expressions.
2. **MapViewAnnotation with SwiftUI**: Use `ForEvery` with `MapViewAnnotation` for individual crimes (simpler but potentially heavier with many points).

**Recommended approach:** Use `SymbolLayer` with pre-registered images for performance. SF Symbols can be rendered to UIImage and registered with the map style.

```swift
// Register images once (in a modifier or onStyleLoaded):
let config = UIImage.SymbolConfiguration(pointSize: 14, weight: .bold)
if let img = UIImage(systemName: "bolt.shield.fill", withConfiguration: config) {
    try? proxy.map?.style.addImage(img, id: "crime-violent")
}
// ... repeat for each category

// Then in SymbolLayer:
var layer = SymbolLayer(id: "crime-single-icons", source: "crime-clusters")
layer.filter = Exp(.not) { Exp(.has) { "point_count" } }
layer.iconImage = .expression(Exp(.match) {
    Exp(.get) { "severity" }
    "violent";  "crime-violent"
    "property"; "crime-property"
    "vehicle";  "crime-vehicle"
    "crime-other"  // default
})
layer.iconSize = .constant(1.0)
layer.iconAllowOverlap = .constant(true)
```

### Anti-Patterns to Avoid
- **Using MapViewAnnotation for hundreds of crime points:** SwiftUI views as map annotations are expensive. Use SymbolLayer with registered images for unclustered points. Reserve MapViewAnnotation for < 50 items.
- **Sharing a single GeoJSON source between heatmap and cluster layers:** The heatmap source should NOT have `cluster: true` (it needs raw points). Use two separate GeoJSONSource instances with the same data but different IDs.
- **Recalculating feature collections on every render:** The `crimeIncidentFC` should only recompute when `incidents` changes, not on camera moves. Since it's a computed property on let-bound data, this is already handled.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Point clustering | Custom grid cell merging | `GeoJSONSource.cluster = true` | Mapbox handles centroid calculation, zoom-level merging, and `point_count` automatically |
| Severity aggregation | Manual loop through visible points | `clusterProperties` with `Exp(.accumulated)` | Server-side aggregation, updates automatically as clusters merge/split |
| Cluster leaf querying | Store incident arrays per grid cell | `getGeoJsonClusterLeaves()` | Handles pagination, returns GeoJSON features directly |
| Heatmap rendering | Custom tile renderer | `HeatmapLayer` with `heatmapWeight` expression | GPU-accelerated, handles density interpolation and zoom transitions |
| Category string matching | Giant switch statement | Keyword-based classifier function | Category strings vary across SF/Oakland/Marin APIs; keyword matching is more robust |

**Key insight:** The existing `DensityGrid`-based approach manually does what Mapbox clustering does natively. Switching to `GeoJSONSource.cluster = true` eliminates ~100 lines of grid code while getting better centroid placement and smoother zoom transitions.

## Common Pitfalls

### Pitfall 1: Heatmap Source Must Not Be Clustered
**What goes wrong:** If you set `cluster: true` on the heatmap's GeoJSON source, the heatmap renders cluster centroids instead of individual incident positions, creating a spotty/blobby appearance instead of smooth heat.
**Why it happens:** Clustering merges nearby points; heatmap needs all raw points for smooth interpolation.
**How to avoid:** Use TWO separate GeoJSONSource instances with different IDs but the same feature collection data. One with `cluster: false` (for heatmap), one with `cluster: true` (for clusters).
**Warning signs:** Heatmap looks like discrete blobs instead of smooth gradients.

### Pitfall 2: clusterProperties Expression Syntax
**What goes wrong:** `clusterProperties` requires a specific two-element expression format `[operator, map_expression]`. Using the wrong format causes silent failures with no aggregated values.
**Why it happens:** The Mapbox expression DSL in Swift is different from the JSON spec. The `Exp(.accumulated)` wrapper is required.
**How to avoid:** Use the exact pattern: `Exp(.accumulated) { Exp(.sum) {}; Exp(.get) { "weight" } }`
**Warning signs:** Cluster circles show "0" or undefined instead of weighted sums.

### Pitfall 3: TapInteraction Layer ID Must Match Exactly
**What goes wrong:** `TapInteraction(.layer("wrong-id"))` silently does nothing.
**Why it happens:** Layer IDs are strings with no compile-time validation.
**How to avoid:** Use string constants (e.g., `enum CrimeLayerId { static let clusterCircles = "crime-cluster-circles" }`) shared between layer creation and interaction setup.
**Warning signs:** Tapping clusters does nothing, no errors logged.

### Pitfall 4: SF Symbol Image Registration Timing
**What goes wrong:** SymbolLayer references an image ID that hasn't been registered yet, showing nothing.
**Why it happens:** Images must be added to the style BEFORE the SymbolLayer renders. In SwiftUI declarative API, timing is tricky.
**How to avoid:** Register images in an `.onStyleLoaded` modifier or use the `StyleImage` declarative API if available in v11.20.
**Warning signs:** Individual crime markers show empty space instead of icons.

### Pitfall 5: Feature Properties Type Mismatch
**What goes wrong:** Properties extracted from tapped features may be `JSONValue` (not raw Swift types). Accessing `.string` on a `.number` value returns nil.
**Why it happens:** GeoJSON properties are heterogeneous; Mapbox returns them as `JSONValue` enum.
**How to avoid:** Use proper `JSONValue` extraction: `feature.properties?["weight"]?.number` (not `.doubleValue`).
**Warning signs:** Detail sheet shows nil/missing data for some fields.

### Pitfall 6: showCrimeDetails Toggle Coupling
**What goes wrong:** Current code gates cluster visibility on `showCrimeDetails` toggle. If clusters should always show (D-17 says heatmap always visible, clusters on top), the toggle semantics need to change.
**Why it happens:** Legacy behavior from DensityGrid era where "Details" was a separate toggle.
**How to avoid:** Decide whether `showCrimeDetails` controls cluster visibility or if clusters are always-on when crime layer is active. The CONTEXT implies clusters should always show (with heatmap as background).
**Warning signs:** Users see heatmap but no clusters until they toggle "Details".

## Code Examples

### Building Severity-Weighted Feature Collection

```swift
// Source: Adapted from existing crimeHotspotsFC pattern in HFMapView.swift
private var crimeIncidentFC: FeatureCollection {
    let dateFormatter = DateFormatter()
    dateFormatter.dateStyle = .short
    dateFormatter.timeStyle = .short

    return FeatureCollection(features: crimeIncidents.map { incident in
        let severity = CrimeSeverity.from(category: incident.category)
        var f = Feature(geometry: .point(Point(incident.coordinate)))
        f.properties = [
            "weight": .number(severity.weight),
            "severity": .string(severity.key),       // "violent"/"property"/"vehicle"/"other"
            "category": .string(incident.category),    // raw API category string
            "description": .string(incident.description),
            "date": .string(dateFormatter.string(from: incident.date))
        ]
        return f
    })
}
```

### Heatmap with Severity-Weighted Green-to-Red Gradient

```swift
// Source: Mapbox HeatmapLayer docs + D-12 through D-17
HeatmapLayer(id: "crime-heat", source: "crime-incidents")
    .heatmapWeight(Exp(.get) { "weight" })  // severity-weighted
    .heatmapIntensity(Exp(.interpolate) {
        Exp(.linear); Exp(.zoom)
        0; 0.5
        9; 1.5
        14; 2.5
    })
    .heatmapRadius(Exp(.interpolate) {
        Exp(.linear); Exp(.zoom)
        0; 4
        9; 25
        14; 50    // ~500m at z14 — wide spread per D-14
    })
    .heatmapColor(Exp(.interpolate) {
        Exp(.linear)
        Exp(.heatmapDensity)
        0;    "rgba(0,0,0,0)"           // transparent
        0.15; "rgba(34,197,94,0.2)"     // green (minor)
        0.30; "rgba(163,230,53,0.35)"   // lime-green
        0.45; "rgba(250,204,21,0.5)"    // yellow (moderate)
        0.60; "rgba(251,146,60,0.6)"    // orange (severe)
        0.80; "rgba(239,68,68,0.7)"     // red
        1.0;  "rgba(220,38,38,0.8)"     // deep red (high density)
    })
    .heatmapOpacity(0.55)  // D-15: 0.5-0.6, street names readable
```

## Open Questions

1. **Cluster leaves query via MapReader proxy**
   - What we know: `MapboxMap.getGeoJsonClusterLeaves` exists in v11. `MapReader` provides `proxy.map` access.
   - What's unclear: Whether `TapInteraction` provides the cluster feature in the right format for `getGeoJsonClusterLeaves` (needs the `cluster_id` property).
   - Recommendation: Test with a simple tap handler first. If the feature from `TapInteraction` doesn't have `cluster_id`, fall back to `queryRenderedFeatures` at the tap point.

2. **SF Symbol image registration in SwiftUI declarative Map**
   - What we know: SymbolLayer needs pre-registered images. `StyleImage` may be available declaratively.
   - What's unclear: Whether `StyleImage` works in `@MapContentBuilder` context in v11.20.1.
   - Recommendation: Try `StyleImage` first. If not available, use `.onStyleLoaded { }` modifier on Map to register images imperatively via `proxy.map?.style.addImage()`.

3. **showCrimeDetails toggle repurposing**
   - What we know: Current toggle controls cluster visibility. D-17 says heatmap always visible with clusters on top.
   - What's unclear: Should clusters also always be visible, or should the toggle still control their visibility?
   - Recommendation: Make both heatmap and clusters always visible when `selectedCategory == .crime`. Remove the `showCrimeDetails` guard from cluster content. The toggle can be repurposed or removed.
